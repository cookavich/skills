# Codebase Wiki Structure

Reference for writing the wiki. Use on both pass 1 (breadth) and pass 2 (deepen). Adapt the page set to repository size and user scope.

## Examples

Concrete sample outputs for the most-generated pages live in `examples/`. Use these as shape references — match the structure, not the content.

- [`examples/sample-architecture.md`](examples/sample-architecture.md) — system architecture page with ASCII diagram
- [`examples/sample-module-map.md`](examples/sample-module-map.md) — module map with per-area annotations
- [`examples/sample-area.md`](examples/sample-area.md) — fully expanded area page (post-pass-2)

## Pass 1 Output

Each page is either **full quality on pass 1** or **stub on pass 1**.

| Page | Pass 1 | Purpose |
|------|--------|---------|
| `README.md` | Full | Nav hub: system summary, reading paths, **Documentation Discovered**, **Hot-spots**, area list, core-page list |
| `architecture.md` | Full | Major components, boundaries, runtime shape, ASCII architecture diagram |
| `module-map.md` | Full | Directory guide. Each area annotated with commit count (6mo) + TODO/FIXME/HACK count + link to area stub |
| `getting-started.md` | Full | Setup, prerequisites, common commands, generated artifacts, first files to inspect |
| `glossary.md` | Full | Domain terms, acronyms, code names, external system names |
| `_inventory.md` | Full (from script) | Generated by the inventory script. Supporting artifact — link to it from README, do not treat it as the main explanation |
| `conventions.md` | Stub | Libraries used, idioms, style patterns. Expanded on demand |
| `configuration.md` | Stub | Env vars, config files, feature flags, build-time vs runtime config |
| `testing.md` | Stub | Frameworks, layout, fixtures, common commands, visible gaps |
| `operations.md` | Stub | CI/CD, deployment, monitoring, runbooks |
| `runtime-flows.md` | Stub | Request, event, command, job, UI, or background flows |
| `data-flow.md` | Stub | Data sources, transformations, persistence, external API movement |
| `areas/<area>.md` | Stub | One per area; capped by repo size |

Omit any page that would have nothing to say even after expansion.

## Stub Format

Every stub is a Markdown file with frontmatter and a small body:

```markdown
---
status: stub
description: <one-line summary>
---

# <Page Title>

Run `/codebase-wiki <scope>` to expand this page.

## Relevant files

- `path/to/file1.ts`
- `path/to/file2.ts`
```

Rules:

- `status: stub` is the detection field. Pass 2 looks for it.
- `description` is required and is one sentence. Used in the README list so the user can pick what to deepen.
- The body has the title, the run hint, and a list of relevant file paths (curated during pass 1 reading, not a raw `find` dump).
- On expansion, the entire frontmatter block is removed from the file.

## Area Page Template (Post-Expansion)

Cover these in order:

- **Responsibility** — what this area owns.
- **Key files** — repo-relative paths with one-line roles. Use line-anchored citations (`src/auth/session.ts:14-58`) where helpful.
- **Public interfaces** — exports, APIs, hooks, commands, routes, schemas, generated clients.
- **Internal flow** — how work moves through the area. ASCII diagram or numbered prose.
- **Dependencies** — what this area depends on (upstream) and what depends on it (downstream).
- **Tests** — where behavior is verified.
- **Common changes** (optional) — only when source reading reveals clear patterns ("to add a new provider, add file in X and register in Y"). Do not invent.
- **Open questions** (optional) — only when the code leaves genuine uncertainty worth flagging.

## README Pattern

The README is the front door. Use this shape:

```markdown
# Codebase Wiki

<one-paragraph system summary>

## Start Here

- [Getting Started](getting-started.md)
- [Architecture](architecture.md)
- [Module Map](module-map.md)

## Documentation Discovered

- `README.md`, `AGENTS.md`, `docs/adr/0001-foo.md`, ...
- Docs site: mkdocs (`mkdocs.yml`)

## Hot-spots (last 6 months)

- `src/auth/` — 87 commits
- `src/billing/` — 54 commits
- `src/api/` — 33 commits

## Areas

- [auth](areas/auth.md) — Authentication and session management *(stub)*
- [billing](areas/billing.md) — Subscription and invoicing *(stub)*

## Core Pages

- [Conventions](conventions.md) *(stub)*
- [Configuration](configuration.md) *(stub)*
- [Testing](testing.md) *(stub)*
- [Operations](operations.md) *(stub)*
- [Runtime Flows](runtime-flows.md) *(stub)*
- [Data Flow](data-flow.md) *(stub)*

## Supporting

- [Inventory](_inventory.md)
```

The Documentation Discovered section pulls from the inventory's "Repository Documentation" and "Docs Site Configuration" sections. Hot-spots pulls from "Hot-spots by Top-Level Directory." Both are first-class signals — they tell the reader where to look first and which stub pass 2 should expand next.

## Linking Rules

- Relative Markdown links only: `[Architecture](architecture.md)`, `[Auth](areas/auth.md)`.
- Stubs link to `../README.md`. Nothing else.
- Expanded pages link to `../README.md` plus at least one related area or core page.
- Cite source paths in backticks: `src/routes.tsx` for file-level, `src/routes.tsx:42-58` for a specific region.
- Never link to absolute local paths.

## Diagrams (ASCII Only)

Use ASCII art. The wiki is for local reading, not GitHub-rendered display, and ASCII renders identically everywhere.

Box-and-line components:

```
+-----------+      +-----------+
|  Client   | ---> |    API    |
+-----------+      +-----+-----+
                         |
                         v
                   +-----+-----+
                   |    DB     |
                   +-----------+
```

Indented call flow:

```
HTTP request hits src/server.ts
  -> middleware/auth.ts validates session
    -> routes/api/billing.ts handles request
      -> services/billing.ts performs work
        -> db/billing.ts writes
```

Boundaries / boxes around module groups:

```
+----------------------------+
|         Web Tier           |
|  routes/, middleware/      |
+----------------------------+
              |
              v
+----------------------------+
|       Service Tier         |
|  services/, validators/    |
+----------------------------+
              |
              v
+----------------------------+
|        Data Tier           |
|       db/, schema/         |
+----------------------------+
```

Keep diagrams small enough to read in raw Markdown. Always pair with prose naming the source files involved.

## Quality Bar

The wiki is done when a new engineer can answer:

- What problem does this repository solve?
- How does execution start?
- What are the major modules and what does each own?
- What external systems does it depend on?
- Where does data enter, change shape, and leave?
- How do I run, test, build, and deploy it?
- Where should I start for a likely change?

Avoid:

- File-by-file summaries with no synthesis.
- Behavioral claims not grounded in source. Every claim should sit near a backticked source path.
- Copying large source excerpts.
- Documenting generated, vendor, or dependency code as if it were authored here.
- Leaking `.env` values, tokens, private hostnames, or local machine paths.
- Mermaid diagrams. Use ASCII.
- Top-level recipes pages. Recipes live inside area pages as the optional "Common changes" section.
- CODEOWNERS data. Out of scope.
- Maintenance-grade prose. This is a throwaway aid; aim for usefulness now, not surviving six months of refactors.
